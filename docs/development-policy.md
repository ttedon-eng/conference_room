# 개발 운영 정책

## 1. 목적
이 문서는 회의실 예약 시스템 프로젝트의 역할 분리, 티켓 상태 관리, 커밋 및 배포 책임 범위를 정의한다.

## 2. 역할 분리 원칙
### 기획자
- 기획자는 기획만 담당한다.
- 기획자는 정책 문서를 작성하고 업데이트한다.
- 기획자는 코드 작성, 테스트 수행, 배포 작업을 하지 않는다.
- 기획자는 개발 티켓의 요구사항과 우선순위를 정의한다.

### 개발자
- 개발자는 개발과 커밋까지 담당한다.
- 개발자는 티켓 상태를 작업 범위 내에서 변경할 수 있다.
- 개발자는 티켓을 임의로 삭제할 수 없다.
- 개발자는 티켓을 임의로 취소할 수 없다.
- 개발자는 푸시와 배포를 직접 수행하지 않는다.

### 배포 관리자
- 배포 관리자는 푸시와 배포를 담당한다.
- 배포 관리자는 테스트 완료된 산출물을 최종 반영한다.
- 배포 관리자는 완료 처리의 최종 책임을 가진다.

## 3. 테스트 원칙
- 테스트는 개발 티켓의 별도 역할이 아니라 검증 단계다.
- 테스터는 테스트 시나리오 작성과 테스트 수행을 담당할 수 있다.
- 테스터는 개발을 하지 않는다.
- 테스터는 코드를 수정하지 않는다.
- 테스트 결과는 티켓 상태 변경의 근거가 된다.

## 4. 개발 티켓 상태 정책
개발 티켓은 다음 상태를 기준으로 관리한다.

- `생성`
- `개발시작`
- `개발완료`
- `테스트완료`
- `완료처리`
- `취소`

### 상태 의미
- `생성`: 티켓이 최초 등록된 상태
- `개발시작`: 개발 작업이 시작된 상태
- `개발완료`: 개발이 끝나고 커밋까지 완료된 상태
- `테스트완료`: 테스트가 완료되어 검증이 끝난 상태
- `완료처리`: 배포 또는 최종 반영이 끝난 상태
- `취소`: 더 이상 진행하지 않는 상태

### 상태 운영 원칙
- 티켓 상태는 역할에 맞는 담당자만 변경할 수 있다.
- 모든 상태 변경은 이력으로 남겨야 한다.
- 티켓 삭제는 원칙적으로 하지 않는다.
- 잘못 생성된 티켓도 삭제보다 `취소` 처리로 관리한다.
- 취소 사유는 가능한 한 명시한다.

## 5. 티켓 삭제 및 취소 정책
- 개발자는 티켓 삭제를 하면 안 된다.
- 개발자는 티켓 취소를 하면 안 된다.
- 티켓 취소는 승인된 담당자만 수행할 수 있다.
- 티켓 상태를 남기기 위해 삭제 대신 취소를 기본 원칙으로 한다.

## 6. 커밋 및 배포 정책
- 개발자의 역할 범위는 커밋까지다.
- 개발자는 배포를 직접 수행하지 않는다.
- 개발자는 푸시를 직접 수행하지 않는다.
- 배포와 반영은 배포 관리자가 담당한다.

## 7. 완료 기준
- `개발완료`는 코드 구현과 커밋이 끝난 상태를 의미한다.
- `테스트완료`는 테스트 검증이 끝난 상태를 의미한다.
- `완료처리`는 배포 또는 최종 반영까지 끝난 상태를 의미한다.
- 최종 완료는 테스트와 배포가 모두 끝난 뒤에만 가능하다.

## 8. 추가 검토 사항
다음 항목은 별도로 정하면 운영이 더 안정적이다.

- 상태 변경 권한표
- 취소 권한과 사유 필수 여부
- 브랜치 및 커밋 규칙
- 테스트 증적 보관 방식
- 배포 승인 조건
- 긴급 장애 대응 절차

## 9. 상태 변경 권한표
개발 티켓 상태 변경은 아래 원칙을 따른다.

| 변경 대상 | 허용 주체 | 비고 |
|---|---|---|
| `생성` -> `개발시작` | 개발자 | 작업 착수 시점 |
| `개발시작` -> `개발완료` | 개발자 | 커밋까지 완료 시점 |
| `개발완료` -> `테스트완료` | 검증 담당자 | 테스트 결과를 근거로 변경 |
| `테스트완료` -> `완료처리` | 배포 관리자 | 배포 또는 최종 반영 후 |
| 어떤 상태 -> `취소` | 승인된 담당자 | 사유 기록 필수 권장 |

## 10. 취소 정책
- 티켓 취소는 원칙적으로 예외 상황에서만 사용한다.
- 취소 사유는 반드시 남긴다.
- 취소된 티켓은 삭제하지 않고 이력으로 보관한다.
- 개발자는 취소를 요청할 수는 있으나 직접 취소하지 않는다.
- 취소 승인 주체는 프로젝트 운영 규칙에서 별도로 지정한다.

## 11. 브랜치 및 커밋 정책
- 개발자는 작업 단위로 브랜치를 사용한다.
- 커밋 메시지는 티켓 번호와 작업 내용을 포함하는 것을 권장한다.
- 개발자는 커밋까지 책임지고, 푸시는 하지 않는다.
- 브랜치 병합과 배포 반영은 배포 관리자가 담당한다.

## 12. 테스트 정책
- 테스트는 개발과 분리된 검증 단계다.
- 테스트 담당자는 테스트 시나리오를 먼저 작성한다.
- 테스트 담당자는 코드 수정을 하지 않는다.
- 테스트 결과는 재현 절차와 함께 기록한다.
- 스크린샷, 로그, 재현 단계 등 증적을 남기는 것을 권장한다.

## 13. 릴리즈 정책
- 배포는 테스트 완료 후에만 진행한다.
- 완료 처리 전에는 배포 관리자 승인이 필요하다.
- 긴급 배포가 필요한 경우에도 최소한의 승인 절차를 둔다.
- 배포 실패 시 롤백 책임과 절차를 사전에 정한다.
